Overview

The framework is built for the agent era after SDK wrappers.

Earlier frameworks organized prompts, tools, and model calls. AgentWorld assumes the worker below the runtime is already a strong agent with tools, files, sessions, and long execution traces. The problem becomes orchestration: how to control them, route work between them, and keep the whole system replayable.

Controller Layer

Claude Code, Codex, and future operators each need different session handling, tool permission mapping, event parsing, timeout control, and workspace binding.

Operator Surface

The graph never speaks provider-specific dialects. Each node invokes a uniform operator request and receives normalized messages, artifacts, state patches, and execution metrics.

A2A Protocol

Agent-to-agent traffic becomes explicit. Messages, handoffs, tool results, and artifacts move as typed payloads instead of hidden prompt glue.

Graph Runtime

Scheduling, branching, checkpointing, replay, interruption, and trace storage belong to the runtime, not to each operator implementation.

Architecture

One control plane, four hard boundaries.

The system stays understandable because each layer owns a narrow job. Provider complexity stops at the controller. Collaboration semantics live in the protocol. Scheduling stays in the runtime.

01

Controllers drive real agents.

Provider-specific adapters create or resume sessions, map tool policies, parse event streams, preserve raw traces, and translate failures into runtime-safe signals.

02

Operators expose one execution contract.

Each operator receives an objective, a state view, an inbox, artifacts, working directory rules, and session policy. It returns normalized outputs that the graph can route without special cases.

03

A2A protocol keeps collaboration explicit.

Case

A real Claude Code graph already runs through the runtime.

The current smoke flow binds a ClaudeCodeController to one operator, runs a planner node that is allowed to use Read, forwards the result through the graph, and finishes with a reviewer node. The observed run includes tool_call, tool_result, PLAN READY, and REVIEW READY.

Runtime Model

What a production-grade multi-agent runtime needs next.

The execution model is narrow enough to stay implementable and strong enough to grow into a real operating layer for multi-agent systems.

1

Unified operator request

Objective, state slice, inbox, artifacts, working directory, tool policy, and timeout live in one request.

2

Normalized operator result

Status, messages, state patch, artifact outputs, handoffs, metrics, and trace references come back in one shape.

3

Persistent execution state

Checkpoint storage keeps every run resumable and makes interruption, replay, and audit practical instead of aspirational.

4

Composable graph semantics

Edges describe handoffs, branching, and review loops without leaking provider-specific control into the graph definition.

Build

Use strong agents as operators. Keep orchestration as software.
